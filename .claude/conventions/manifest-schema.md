# Service Manifest Schema

Each service has a `manifest.yaml` in `docs/architectures/{service}/manifest.yaml`. This file is the **source of truth** for which conventions the service follows. Skills (`product-create-*-architecture`, `service-planify-story`, `service-implement-story`) consume it.

## Schema

```yaml
service: {service-name}              # required. Service identifier. kebab-case.
type: {api|worker|cli|library|frontend}  # required. Service type.
language: {node|nextjs|...}          # required. Language/runtime. Must match a folder in .claude/conventions/.

# Conventions used by the service.
# _base is implicit and always included for the language.
# Conventions auto-included via `required_by` should NOT be listed here
# (the resolver adds them at read time).
#
# Each id is resolved in order:
#   1. docs/architectures/{service}/conventions/{id}.md  (custom, per-service)
#   2. .claude/conventions/{language}/{id}.md            (catalog)
# If not found in either, the resolver fails.
conventions:
  - {convention-id}
  - {convention-id}

# Domain modules of the service. Used by skills to generate structure and route to code.
modules:
  - {module-id}                      # short. matches src/domain/{module-id}/
```

## Resolution rules

When a skill needs to know "which conventions does this service follow", it resolves the manifest as:

1. **Always include** `_base.md` of the declared `language`.
2. **For each item in `conventions`**:
   - Look up `docs/architectures/{service}/conventions/{id}.md` first.
   - If not present, look up `.claude/conventions/{language}/{id}.md`.
   - If not present in either, fail with a clear error.
3. **For each included convention**, read its frontmatter and check `required_by`. If any **listed** convention is already active, the **current** convention is also active. Iterate until stable (transitive closure).
   - The `required_by` evaluation does not care about origin: custom and catalog conventions participate equally.
   - Example: `http-server` is in the manifest. `error-handling` has `required_by: [http-server, validation, auth-jwt]`. Therefore `error-handling` is included automatically.
4. **Validate**:
   - Every `conventions` entry resolves to an existing file (custom or catalog).
   - Every `applies_to` of each active convention includes the service `type`. Warn if it does not.

## Custom conventions

When the catalog convention does not fit a service, the service defines its own:

```
docs/architectures/{service}/
├── manifest.yaml
├── overview.md
├── index.md
└── conventions/
    └── {convention-id}.md          # custom convention, same format as catalog
```

A custom convention:
- Uses the **same frontmatter format** as catalog conventions (id, language, description, applies_to, required_by, package).
- Is referenced in the manifest the same way (just the id in `conventions:`).
- Can declare `required_by` to auto-include itself based on other active conventions.
- Belongs only to its service. Other services that need the same thing define their own.

If a custom convention has the **same id** as a catalog convention, the custom **wins** (resolver checks custom path first). This is the mechanism to replace a catalog convention for one service: create a file with the same id and modify what you need.

## Example: API service using only catalog conventions

```yaml
service: user-service
type: api
language: node

conventions:
  - http-server
  - validation
  - logging
  # error-handling is auto-included (required_by http-server)

modules:
  - users
  - sessions
  - tokens
```

Resolves to active conventions: `_base`, `http-server`, `validation`, `logging`, `error-handling` (all from catalog).

## Example: Service with a custom convention

```yaml
service: reports-service
type: api
language: node

conventions:
  - http-server
  - validation
  - logging
  - pdf-rendering        # custom, defined at docs/architectures/reports-service/conventions/pdf-rendering.md

modules:
  - reports
  - templates
```

`pdf-rendering` is found at the custom path; the rest at the catalog.

## Example: Service replacing a catalog convention

```yaml
service: legacy-api
type: api
language: node

conventions:
  - http-server         # resolves to docs/architectures/legacy-api/conventions/http-server.md
                        # (the file exists locally and takes precedence over the catalog)
  - validation
  - logging

modules:
  - reports
```

The local file documents the replacement (e.g., uses Express instead of Fastify with all the rules adapted to that variant). It has the same id, the manifest does not change shape.

## What lives where

| File | Location | Generated by | Editable by hand? |
|---|---|---|---|
| `manifest.yaml` | `docs/architectures/{service}/` | `product-create-*-architecture` | yes |
| `overview.md` | `docs/architectures/{service}/` | `product-create-*-architecture` | yes |
| `index.md` | `docs/architectures/{service}/` | auto-generated from manifest | no (regenerated) |
| Custom conventions | `docs/architectures/{service}/conventions/` | `product-change-technical-definition` | yes |
| Catalog conventions | `.claude/conventions/{language}/` | maintained in workflow repo | only in workflow repo |
