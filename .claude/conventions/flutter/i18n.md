# Internationalization (Flutter)

User-facing strings are externalized per language and resolved through a translation layer. No literal user-facing text in widgets. Language can switch at runtime.

## When to use

Any app shipping in more than one language, or that may in the future. Default: per-language resource files (JSON or ARB) + a translation accessor. Single-language apps still externalize strings so adding a language later is cheap.

## Structure

```
lib/shared/i18n/
├── translations/
│   ├── en.json
│   ├── es.json
│   └── pt.json
├── translation_service.dart   # loads files, resolves keys, fallback
└── l10n_extension.dart         # context.t('key') helper
```

Declare the translation assets in `pubspec.yaml`.

## How to use

### Resource files

```json
// lib/shared/i18n/translations/es.json
{
  "orders.title": "Pedidos",
  "orders.empty": "No hay pedidos",
  "pairing.code": "Tu código: {code}"
}
```

### Resolving a key

```dart
Text(context.t('orders.title'));
Text(context.t('pairing.code', {'code': code}));   // interpolation
```

### Runtime language switch

The current language is a provider; changing it rebuilds the tree (the translation service exposes a version/notifier that widgets watch — see `state-management`). Detect the initial language from the platform, allow a manual override, and persist the choice.

## Rules

- No user-facing literal strings in widgets. Every visible string is a translation key.
- Keys are namespaced by feature (`orders.title`, `pairing.code`). Flat, descriptive, stable.
- Every key exists in **all** language files. A missing key falls back to the default language (and is flagged in dev), never shows the raw key in production.
- Interpolation uses named placeholders (`{code}`), not string concatenation.
- Locale-aware formatting (dates, numbers, currency) uses `intl`, not manual formatting. Dates remain UTC at the boundary (see `_base`).
- Proper names are not translated; UI labels, categories, and messages are.
- Adding a language = adding a resource file + registering it. No code changes in widgets.

## Variant

`flutter_gen-l10n` (ARB + generated accessors) or [slang](https://pub.dev/packages/slang) (type-safe keys via codegen) are acceptable and give compile-time safety over raw JSON keys. The rules (externalized strings, namespaced keys, full coverage, `intl` formatting, runtime switch) apply identically. New apps may prefer a type-safe option. The current app also uses a JSON + service approach, but with **nested** keys (e.g. `screens.modeSelection.title`) resolved by a `t(context, key)` free function and a `ref.t(key)` extension — rather than the flat `context.t('orders.title')` shown above. Same idea, different key shape and accessor.

## Integration with other conventions

- **state-management**: the active language is a provider; widgets watch it to rebuild on switch.
- **theming**: text styles come from the design system; i18n only supplies the content.
