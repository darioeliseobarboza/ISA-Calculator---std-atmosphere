# ORM / Database Access (Node, Prisma)

Typed database access with [Prisma](https://www.prisma.io). Schema declared in `prisma/schema.prisma`, queries through the Prisma Client. All DB access is encapsulated behind a repository layer — no direct Prisma Client calls from handlers or use-cases.

## When to use

- Any Node service that reads or writes a relational database (PostgreSQL, MySQL, SQLite).
- APIs, workers, and CLIs that need persistence.
- Not applicable when the service has no database (e.g., a proxy or pure-in-memory service).

## Package

```
prisma              # dev only — CLI for schema management and migrations
@prisma/client      # runtime — generated client
```

## Structure

```
prisma/
├── schema.prisma       # data model and migration config
└── migrations/         # generated by prisma migrate
src/
├── db/
│   └── client.ts       # PrismaClient singleton
└── domain/
    └── {module}/
        └── {module}.repository.ts   # repository per domain module
```

## Configuration

```ts
// src/db/client.ts
import { PrismaClient } from '@prisma/client';

declare global {
  // eslint-disable-next-line no-var
  var prisma: PrismaClient | undefined;
}

// Singleton to avoid exhausting connections in development (hot reload)
export const prisma = global.prisma ?? new PrismaClient({
  log: process.env.NODE_ENV === 'development' ? ['query', 'warn', 'error'] : ['warn', 'error'],
});

if (process.env.NODE_ENV !== 'production') global.prisma = prisma;
```

```prisma
// prisma/schema.prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model User {
  id        String   @id @default(uuid())
  email     String   @unique
  createdAt DateTime @default(now())
}
```

## How to use

### Repository pattern

One repository class per domain module. It receives `prisma` via constructor injection.

```ts
// src/domain/users/users.repository.ts
import type { PrismaClient, User } from '@prisma/client';

export class UsersRepository {
  constructor(private prisma: PrismaClient) {}

  async findById(id: string): Promise<User | null> {
    return this.prisma.user.findUnique({ where: { id } });
  }

  async create(data: { email: string }): Promise<User> {
    return this.prisma.user.create({ data });
  }

  async update(id: string, data: Partial<{ email: string }>): Promise<User> {
    return this.prisma.user.update({ where: { id }, data });
  }
}
```

### Transactions

Use `prisma.$transaction` when coordinating multiple writes:

```ts
await prisma.$transaction(async (tx) => {
  const repo = new UsersRepository(tx as unknown as PrismaClient);
  await repo.create({ email: 'a@b.com' });
  // ... other writes in the same transaction
});
```

For interactive transactions (reads between writes), use `$transaction` with the callback form above. For simple batching, use the array form:

```ts
await prisma.$transaction([
  prisma.user.create({ data: { email: 'a@b.com' } }),
  prisma.order.create({ data: { userId: '...' } }),
]);
```

### Migrations

```bash
# create and apply a migration in development
npx prisma migrate dev --name add_users_table

# apply pending migrations in production
npx prisma migrate deploy

# regenerate the Prisma Client after schema changes
npx prisma generate
```

Run `prisma migrate deploy` at service startup before the HTTP server starts:

```ts
// src/db/migrate.ts
import { execSync } from 'child_process';

export function runMigrations() {
  execSync('npx prisma migrate deploy', { stdio: 'inherit' });
}
```

## Rules

- No direct `prisma` calls from HTTP handlers or use-case functions. Always go through a repository.
- One repository per domain module. Never share a repository across unrelated modules.
- Use Prisma-generated types (`User`, `Order`, etc.) directly — no manual type declarations that duplicate the schema.
- `DATABASE_URL` is read from env by Prisma itself (`env("DATABASE_URL")` in schema.prisma). Never hardcode it.
- Migrations run automatically at startup in production via `prisma migrate deploy`. Never run `migrate dev` in prod.
- Commit both `prisma/schema.prisma` and the `prisma/migrations/` folder. The migrations folder is the source of truth for DB history.
- Run `prisma generate` after every schema change during development. CI must also run it before type checking.
- Do not use `prisma.$queryRaw` unless Prisma cannot express the query. If raw SQL is needed, document why.

## Integration with other conventions

- **env-config**: `DATABASE_URL` is validated at startup by the env config convention.
- **error-handling**: repository methods let Prisma errors propagate; the service layer catches and converts to domain errors. `PrismaClientKnownRequestError` with code `P2025` maps to a `NotFoundError`.
- **testing**: repositories are integration-tested against a real DB (Testcontainers) — not mocked.
