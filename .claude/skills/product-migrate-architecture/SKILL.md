# Migrate Architecture

## Purpose

Migrate one or more services from the legacy architecture format (multiple section files: `tech-stack.md`, `project-structure.md`, etc.) to the new format (`manifest.yaml` + `overview.md` + `index.md` + custom conventions in `docs/architectures/{service}/conventions/`).

**Flow:**
```
Step 0: Detect services to migrate, ask for service repo path(s)
Step 1: Launch Technical Leader subagent in background (per service)
        — reads source code + legacy docs
        — produces structured draft: _base + one convention per concern
        — infers purpose and domain modules
Step 2: Present full draft to user
        --- GATE A: user confirms, adjusts, or discards conventions ---
Step 3: Generate convention files + manifest.yaml + overview.md + index.md
        Delete legacy files
        --- GATE B: user reviews generated files ---
Step 4: Summary
```

**This skill is interactive only.** It does not accept `--auto`.

**Result:**
- `docs/architectures/{service}/manifest.yaml`
- `docs/architectures/{service}/overview.md`
- `docs/architectures/{service}/index.md`
- `docs/architectures/{service}/conventions/{id}.md` — one per detected concern
- Legacy section files removed

**This command does NOT:**
- Create architecture from scratch — use `/product-create-backend-architecture` or `/product-create-frontend-architecture`
- Modify conventions after migration — use `/product-change-technical-definition`
- Touch the global conventions catalog (`.claude/conventions/`)

## Role

**Adopt the Technical Leader Agent role** — Read [Technical Leader Agent](.claude/agents/technical-leader.md)

## CRITICAL RULES

1. **MANDATORY CONFIRMATION GATES.** Two gates (A and B). Each is a hard stop — call `AskUserQuestion`, wait for response, do not proceed until the user answers. No exceptions.

2. **All user-facing text in Spanish.** The skill file is in English (for the agent); every word the user sees is in Spanish.

3. **Source of truth is the code, not the docs.** The legacy documentation is context. The actual package.json and source files determine what conventions to create. If docs say "we use Pino" but the code uses Winston, go with Winston.

4. **Custom conventions only.** Do NOT reference or suggest catalog conventions (`.claude/conventions/`). Every convention generated by this skill goes into `docs/architectures/{service}/conventions/`. The manifest lists only custom convention ids.

5. **Convention granularity rule.** Create a convention for each concern that:
   - Has its own configuration setup
   - Has recurring usage patterns across the codebase
   - A developer needs to know to write consistent code
   Do NOT create conventions for: utility libraries with no particular pattern, single-use packages, internal helpers.

6. **Always include `_base`.** Every service gets a `_base` convention covering general patterns: naming, project structure, TypeScript config, import style, date handling, async patterns — inferred from the actual code.

7. **Custom conventions follow the Convention Template.** Read [Convention Template](.claude/templates/convention-tmpl.yaml) before generating any file. The frontmatter format, mandatory sections, and quality checklist apply.

8. **Delete legacy files only after GATE B approval.** Never delete before the user has reviewed the generated files.

9. **Do NOT dump full content in chat.** After generating files, show a summary with paths. Let the user review files directly.

10. **Reference locations from Files index.** Do not hardcode paths.

## Execution

### Step 0: Detect Services and Get Repo Paths

**0.1 Read Files index**

Read [Files index](.claude/utils/index.md) → identify **architectures_folder**.

**0.2 Detect legacy services**

Scan `docs/architectures/` for services in legacy format: directories that contain `tech-stack.md` but do NOT contain `manifest.yaml`.

If `$ARGUMENTS` provided: check only that service. If not found or not legacy, inform the user and abort.

If no argument: find all legacy services.

If no legacy services found:

```markdown
No se encontraron servicios en formato legacy.

Todos los servicios ya tienen `manifest.yaml`. No hay nada que migrar.
```

**ABORT.**

**0.3 Inform user and ask for repo paths**

```markdown
## Servicios a migrar

Se encontraron {N} servicio(s) en formato legacy:

{for each service:}
- **{service-name}** — `docs/architectures/{service}/`

Para migrar, necesito acceso al código fuente de cada servicio.
{if N == 1:}
¿Dónde está el repositorio de **{service-name}**? (ruta absoluta o relativa)
{else:}
Para cada servicio, indicá la ruta al repositorio (ruta absoluta o relativa).
Podés responder en una lista:
- {service-1}: /ruta/al/repo
- {service-2}: /ruta/al/repo
```

Wait for response. Store as `repo_paths: { service-name: path }`.

Verify each path exists and contains a `package.json`. If any path is invalid, inform the user and ask to correct before continuing.

---

### Step 1: Launch Subagent per Service

For each service to migrate, launch a subagent in background with this prompt:

---

**Subagent prompt:**

```
You are a Technical Leader Agent. Read the file .claude/agents/technical-leader.md for your role.

Your task: analyze the source code and legacy architecture docs of the service "{service-name}" and produce a structured migration draft.

**Inputs to read:**
1. Legacy architecture docs at: docs/architectures/{service-name}/ (read all .md files present)
2. Source code at: {repo_path}
   - package.json (dependencies and devDependencies)
   - tsconfig.json if present
   - Source files — read enough to understand patterns: entry point, a representative module or two, error handling setup, logging setup, validation setup, middleware/route setup, database access layer
   - Test files — read at least one representative test file to understand the testing framework, setup/teardown patterns, assertion style, fixture approach, and test structure

**Your output must be a structured text block with this exact format:**

---MIGRATION DRAFT: {service-name}---

SERVICE TYPE: {api | worker | cli | library}
LANGUAGE: {node | nextjs | ...}
PURPOSE: {one sentence inferred from the code and docs}
MODULES: {comma-separated list of domain modules inferred from folder structure}

CONVENTIONS:
---
ID: _base
DISPLAY NAME: {short Spanish name, e.g. "Convenciones generales"}
PACKAGE: null
CONCERN: General code patterns
SECTIONS:
- TypeScript: {tsconfig strictness, key compiler options observed}
- Naming: {file naming pattern, class/function naming observed}
- Project structure: {actual folder layout observed}
- Imports: {import style observed — aliases, ordering}
- Async: {async/await vs callbacks observed}
- Dates: {how dates are handled in the code}
RULES:
1. {concrete rule inferred from the code}
2. {concrete rule}
...
---
ID: {concern-id}
DISPLAY NAME: {short Spanish name, include package in parens if identity-defining}
PACKAGE: {package name}
CONCERN: {e.g. HTTP server, Logging, Validation, ORM, Error handling, Auth}
SECTIONS:
- Configuration: {how the package is configured/initialized}
- Usage patterns: {how it is used across the codebase}
- Key rules observed: {implicit rules the code follows}
RULES:
1. {concrete rule inferred from the code}
2. ...
---
{repeat for each concern found}

LEGACY FILES TO DELETE:
- {filename}
- {filename}
...

---END DRAFT---

Rules for deciding which concerns deserve a convention:
- Must have configuration setup + recurring usage pattern + something a new developer needs to know to write consistent code
- The concern must be TECHNICAL (a cross-cutting pattern), not domain-specific (business rules of this particular application)
- DO NOT create conventions for: utility libraries with no pattern, single-use packages, internal helpers, domain-specific business logic (e.g. "vehicle assignment rules", "roadmap stage completion") — those belong in the service's own documentation, not in architecture conventions
- ALWAYS include _base
- Common concerns to look for: http-server, logging, validation, error-handling, orm/database, auth, testing, messaging/queue, cache, observability, scheduled-tasks, file-storage

A good convention answers "how do we use this technology across the codebase?" — not "what are the business rules of this application?".

Be concrete. Rules must be imperative and verifiable ("use kebab-case for filenames", "never log passwords"). 
Aim for 5-15 rules per convention.
Output ONLY the draft block. No preamble, no explanation.
```

---

Notify the user while waiting:

```markdown
Analizando el código de **{service-name}**... esto puede tardar unos segundos.
```

If multiple services: launch all subagents in parallel, notify once for all.

Wait for all subagents to complete before proceeding.

---

### Step 2: Present Draft and Confirm

Parse each subagent response to extract the draft.

Render for each service:

```markdown
## Borrador de migración: **{service-name}**

**Tipo:** {service_type} | **Lenguaje:** {language}
**Propósito inferido:** {purpose}
**Módulos:** {modules}

### Convenciones detectadas

| Convención | Paquete | Preocupación |
|---|---|---|
| {display_name} | {package or —} | {concern} |
| ... | ... | ... |

### Reglas por convención

**{display_name} (_base)**
1. {rule}
2. {rule}
...

**{display_name} ({id})**
1. {rule}
2. {rule}
...

### Archivos legacy que se eliminarán
{list of legacy files}
```

**=== GATE A — Confirm draft ===**

Ask with `AskUserQuestion`:
- Question: "¿El borrador es correcto?"
- Options:
  - `Confirmar` / "Genero los archivos con este borrador"
  - `Ajustar` / "Quiero corregir algo"

**STOP and wait.**

If `Ajustar`: ask free-text what to change (add/remove a convention, correct a rule, fix purpose/modules). Update the in-memory draft, re-render Step 2, re-open GATE A. Loop until confirmed.

---

### Step 3: Generate Files

For each service, execute in order:

**3.1 Read Convention Template**

Read [Convention Template](.claude/templates/convention-tmpl.yaml). Keep frontmatter spec, mandatory sections, and quality checklist in memory.

**3.2 Generate custom convention files**

For each convention in the approved draft, write `docs/architectures/{service}/conventions/{id}.md`:

- Frontmatter: all mandatory fields. `language` from draft. `applies_to` from service type. `required_by: []`. `package` from draft.
- Body: build from the SECTIONS and RULES in the draft, following the convention template structure.
- `_base`: use the special `_base` structure (multiple H2 sections by topic instead of a single `## How to use`).
- Other conventions: standard structure — intro, `## When to use`, `## Package`, `## Configuration`, `## How to use`, `## Rules`, plus optional sections as needed.
- Content in English. `display_name` in Spanish.
- Apply quality checklist before saving.

**3.3 Generate manifest.yaml**

Write `docs/architectures/{service}/manifest.yaml`:

```yaml
service: {service-name}
type: {service_type}
language: {language}

conventions:
  - _base
  - {id}
  - {id}
  ...

modules:
  - {module-id}
```

All conventions are custom, so all ids go into the manifest explicitly.

**3.4 Generate overview.md**

Write `docs/architectures/{service}/overview.md` in Spanish:

```markdown
# {service-name} — Overview

## Propósito
{purpose}

## Tipo de servicio
{service_type} en {language}

## Módulos del dominio
- **{module-id}**
- ...

## Notas adicionales
Sin notas adicionales por el momento.

---

**Manifest:** [manifest.yaml](./manifest.yaml)
**Índice de arquitectura:** [index.md](./index.md)
```

**3.5 Generate index.md**

Write `docs/architectures/{service}/index.md` in Spanish:

```markdown
# {service-name} — Arquitectura

> Auto-generado desde [manifest.yaml](./manifest.yaml). No editar a mano.

**Tipo:** {service_type} | **Lenguaje:** {language}

## Específico del servicio
- [Overview](./overview.md)
- [Manifest](./manifest.yaml)

## Convenciones activas

- **[{display_name}](./conventions/{id}.md)** — {description} (custom)
- ...

---

**Total:** {N} convenciones activas.
```

**3.6 Delete legacy files**

Delete each file listed in the draft's `LEGACY FILES TO DELETE` section.

**3.7 Notify user**

```markdown
Migración de **{service-name}** completada.

**Archivos generados:**
- `docs/architectures/{service}/manifest.yaml`
- `docs/architectures/{service}/overview.md`
- `docs/architectures/{service}/index.md`
- Convenciones custom ({N}):
{for each:}
  - `docs/architectures/{service}/conventions/{id}.md` — {display_name}

**Archivos legacy eliminados:** {N}

Revisá los archivos y decime si están correctos o querés cambios.
```

**=== GATE B — Review generated files ===**

Ask with `AskUserQuestion`:
- Question: "¿Los archivos generados están correctos?"
- Options:
  - `Aprobar` / "Todo bien, continúa"
  - `Pedir cambios` / "Quiero corregir algo"

**STOP and wait.**

If `Pedir cambios`: ask free-text what to change. Apply edits to the relevant files. Re-open GATE B. Loop until approved.

---

### Step 4: Summary

```markdown
## Migración completada

{for each migrated service:}
### {service-name}
- **Convenciones generadas:** {N}
- **Manifest:** `docs/architectures/{service}/manifest.yaml`
- **Archivos legacy eliminados:** {N}

---

**Siguientes pasos:**
- Los próximos planes de stories van a leer las convenciones custom automáticamente
- Si querés ajustar convenciones más adelante: `/product-change-technical-definition {service-name}`
- Si querés planificar una story: `/service-planify-story S-XXX`
```

## Output

Files created per migrated service:

- `docs/architectures/{service}/manifest.yaml`
- `docs/architectures/{service}/overview.md`
- `docs/architectures/{service}/index.md`
- `docs/architectures/{service}/conventions/_base.md`
- `docs/architectures/{service}/conventions/{id}.md` (one per detected concern)

Legacy section files deleted.
