# Story Plan: S-002 (atmosphere-app) - Esqueleto atmosphere-app + pantalla de health

## Status

**Status:** Ready

## Referencia de la Story

- **Story:** [S-002](../stories/S-002.app-skeleton-health.md)
- **Servicio:** atmosphere-app
- **Resumen:** Crear el esqueleto del frontend Flutter `atmosphere-app` (targets Windows, Linux y Web) con configuración por entorno (`API_BASE_URL` vía `flutter_dotenv`, fail-fast), un `ApiClient` (package `http`) + repository de health, manejo de errores de dominio, theming/navegación base y una pantalla **provisional** de health que consume `GET /health` y muestra si el sistema está vivo o en error de conexión. ## Contexto Arquitectónico

> Esta sección es autocontenida: el contenido técnico se copia aquí para que el implementador no necesite abrir otros documentos del producto.

### Stack Tecnológico

- **Lenguaje:** Flutter / Dart (última estable; pinear el SDK en `pubspec.yaml` → `environment: sdk` y en CI). Sound null safety en todo el código (sin `// @dart` opt-outs).
- **Estado / DI:** [`flutter_riverpod`](https://riverpod.dev) — `StateNotifier` + providers; Riverpod también es el mecanismo de DI (servicios/repositories se proveen, no se instancian inline). NO usar `get_it`.
- **Networking:** package oficial [`http`](https://pub.dev/packages/http) envuelto por un `ApiClient` fino + un repository por dominio (custom de servicio, reemplaza el default `dio` del catálogo, por ADR-001).
- **Routing:** [`go_router`](https://pub.dev/packages/go_router) — tabla de rutas única, navegación por URL.
- **Config de entorno:** [`flutter_dotenv`](https://pub.dev/packages/flutter_dotenv) — config tipada cargada al arranque (fail-fast si falta una variable requerida).
- **Formato / i18n:** [`intl`](https://pub.dev/packages/intl) — aplica plenamente en FG-2 (formato numérico de la calculadora). En esta story es mínimo (la UI es una pantalla provisional); no es bloqueante.
- **Testing:** `flutter_test` (SDK) + [`mocktail`](https://pub.dev/packages/mocktail) (mocks sin codegen). `test/` espeja `lib/`.
- **Formateo:** `dart format` (line length 80 salvo que el proyecto indique otra). Código sin formatear no mergea. El analyzer corre limpio (warnings no se ignoran).
- **Persistencia:** ninguna — la app solo lee `/health` y (en FG-2) consume `POST /v1/calculate`. No persiste estado.

- **`screens/{feature}/`**: una pantalla + los widgets usados **solo** por ella (en su `widgets/`). Lo reutilizable va a `shared/widgets/`.

### ADRs Relevantes

#### ADR-001: Stack y arquitectura cliente-servidor

- **Decisión:** `atmosphere-app` es Flutter (Dart) con Riverpod (estado), `http` (cliente) e `intl` (formato), targets `windows`/`linux`/`web`. Comunicación HTTP/REST JSON síncrona. Sin autenticación en v1; CORS habilitado del lado API.
- **Implementation Rules (verbatim, las que aplican a esta story):**
  - El frontend MUST ser Flutter (Dart) con **Riverpod** (estado), **`http`** (cliente) e **`intl`** (formato). El frontend NO calcula: solo presenta lo que devuelve la API.
  - La comunicación MUST ser HTTP/REST JSON síncrona; no hay procesos async ni mensajería.
  - Respuestas HTTP: `200` éxito; `400` validación con cuerpo `{ "error": { "code": string, "message": string } }`, con `code` ∈ `outOfRange` | `invalidInput` | `invalidStep`. *(No aplica al health check, pero define el shape de error que `ApiClient` debe mapear para FG-2.)*
  - Los identificadores JSON MUST ir en **inglés** (tipos `PascalCase`, campos/enums `camelCase`).
  - Las rutas de la API MUST llevar prefijo de versión **`/v1/`** (excepto `/health`, que va sin prefijo).

#### ADR-003: Servicio stateless (sin persistencia)

- **Decisión:** `atmosphere-api` es stateless. Implicación para la app: no hay sesión ni token; la app solo hace requests idempotentes de lectura.
- **Implementation Rules (verbatim, las que aplican):**
  - `atmosphere-api` MUST ser stateless: NO base de datos, NO persistencia en disco, NO estado de sesión entre requests.
  - *(Consecuencia para la app: el health check no requiere auth ni manejo de sesión; `go_router` no necesita guards de sesión en esta story.)*

#### ADR-004: Estrategia de deployment (monorepo + Docker dev/prod + ingress-network)

- **Decisión:** Monorepo `services/atmosphere-api`, `services/atmosphere-app`, `docker/`. El target web se sirve por un contenedor nginx (`atmosphere-web`); los targets desktop son binarios nativos (no contenerizados). CORS habilitado del lado API.
- **Implementation Rules (verbatim, las que aplican a esta story):**
  - `atmosphere-web` MUST ser un contenedor nginx que sirve el build de Flutter web; los targets desktop (Windows/Linux) MUST ser binarios nativos (`flutter build windows`/`linux`), NO contenerizados.
  - CORS MUST estar habilitado en `atmosphere-api` con orígenes permitidos vía `CORS_ALLOWED_ORIGINS` por ambiente. *(La app NO maneja CORS; es responsabilidad de la API. La app solo debe poder llamar a un origen distinto, lo cual `http` hace nativamente.)*
  - Variables de entorno: `atmosphere-web` → `API_BASE_URL`. *(Esta es la variable que la app consume vía `flutter_dotenv`.)*
  - **Nota de alcance:** los archivos `docker/docker-compose*.yml` (base + overrides) son de **S-003** (orquestación), NO de esta story. Esta story solo crea el `Dockerfile` del target web y documenta el build desktop.

### Contexto de API (endpoint consumido)

La app **consume** (no implementa) el endpoint `GET /health` de `atmosphere-api`. Contrato exacto desde `docs/apis/atmosphere-api.yaml`:

**Request:**
- **Método:** `GET`
- **Endpoint:** `{API_BASE_URL}/health` (sin prefijo `/v1/`)
- **Headers:** ninguno requerido (sin auth)

**Response (éxito):**
- **Status:** `200`
- **Body:**
  ```json
  {
    "status": "ok",
    "timestamp": "2026-06-12T16:00:00Z"
  }
  ```
  - `status`: `string` (ejemplo: `"ok"`)
  - `timestamp`: `string` formato `date-time` (ejemplo: `"2026-06-12T16:00:00Z"`)

**Manejo de errores (desde el flujo de la story):**
- Timeout / sin conectividad / status `≥ 400` → la app muestra **estado de error de conexión** (NO "vivo"). No hay reintentos automáticos en esta story (un ping al abrir).

> El cuerpo de error `{ "error": { "code", "message" } }` (ADR-001) NO aplica a `/health` (que solo devuelve 200), pero `AppException.fromResponse` debe construirse genérico para reutilizarlo en FG-2 con `/v1/calculate`.

### Contexto de Flujo

**Flujo de la story (health check end-to-end)** — definido inline en la story S-002 (no modifica ningún flujo documentado en `docs/flows/`; `calculate.md` es FG-2 y queda fuera de alcance):

1. **[atmosphere-app]** El usuario abre la app → `HealthNotifier.checkHealth()` → `HealthRepository.check()` → `ApiClient` hace `GET {API_BASE_URL}/health`.
2. **[atmosphere-api]** Responde `200 { "status": "ok", "timestamp": ... }`.
3. **[atmosphere-app]** Mapea a estado `alive` y la pantalla muestra "API disponible / sistema vivo".

**Camino de error:** timeout / sin red / status ≥ 400 → `ApiClient` lanza `AppException` → `HealthNotifier` captura y pone estado `error` → la pantalla muestra "error de conexión".

**Contexto cross-origin (CA-4):** en web, la app y la API están en orígenes distintos. La llamada `GET /health` debe funcionar sin error de CORS. La habilitación de CORS es responsabilidad de `atmosphere-api` (ADR-004); del lado de la app no hay nada especial que hacer salvo no asumir mismo origen (la base URL viene de `API_BASE_URL`, nunca relativa/hardcodeada).

### Integration Points

- **atmosphere-api (servicio interno):** único punto de integración. Endpoint `GET /health` (ver Contexto de API). Sin auth, sin headers especiales. Base URL desde `API_BASE_URL`.
- **Sin colas/eventos:** comunicación 100% HTTP request/response síncrona (ADR-001).

### Convenciones de Código (resumen accionable)

**Networking (`http` + repository) — reglas verbatim:**
- Notifiers/widgets dependen de una **interface de repository**, nunca de `http.Client`/`ApiClient` directamente. El cliente es detalle de implementación detrás del repository.
- Un repository por dominio, provisto vía Riverpod.
- Base URL, timeouts y headers vienen del **`ApiClient`** + `env-config` — nunca hardcodeados por llamada (`http` no tiene interceptores, así que el wrapper los centraliza).
- Los repositories devuelven **modelos de dominio**; el mapeo DTO→dominio ocurre en esa capa. Los tipos de transporte no se filtran a la UI.
- Los errores de transporte se mapean a `AppException` de dominio **en el `ApiClient`/repository** según status + body. Los widgets manejan errores de dominio, no excepciones crudas de `http`.
- Cada request tiene un timeout explícito (`.timeout(...)`). Cancelar requests en vuelo en `dispose` cuando aplique.
- Nunca loguear respuestas completas ni headers de auth.

**ApiClient de referencia (de la convención `networking`):**
```dart
// lib/shared/services/api_client.dart
final apiClientProvider = Provider<ApiClient>((ref) {
  final env = ref.read(envProvider);
  return ApiClient(baseUrl: env.apiBaseUrl, inner: http.Client(), timeout: const Duration(seconds: 15));
});

class ApiClient {
  ApiClient({required this.baseUrl, required http.Client inner, required this.timeout}) : _inner = inner;
  final String baseUrl;
  final Duration timeout;
  final http.Client _inner;

  Future<Map<String, dynamic>> getJson(String path) async {
    final res = await _inner
        .get(Uri.parse('$baseUrl$path'))
        .timeout(timeout);
    final decoded = res.body.isEmpty ? <String, dynamic>{} : jsonDecode(res.body) as Map<String, dynamic>;
    if (res.statusCode >= 400) {
      throw AppException.fromResponse(res.statusCode, decoded);
    }
    return decoded;
  }
  // postJson(...) similar — se agrega en FG-2 para /v1/calculate
}
```
> La convención muestra `postJson`; para esta story se necesita `getJson` (mismo patrón: timeout explícito, decodificación segura de body vacío, mapeo de status ≥ 400 a `AppException`). Errores de transporte (sin red / timeout) deben capturarse y traducirse a `NetworkException`.

**Manejo de errores (`error-handling`) — modelo de dominio:**
```dart
// lib/shared/errors/app_exception.dart
sealed class AppException implements Exception {
  const AppException(this.message);
  final String message;
}
class NetworkException extends AppException {        // sin conectividad / timeout
  const NetworkException([super.message = 'network error']);
}
class NotFoundException extends AppException { const NotFoundException([super.message = 'not found']); }
class UnauthorizedException extends AppException { const UnauthorizedException([super.message = 'unauthorized']); }
class ValidationException extends AppException { const ValidationException(super.message, this.fields); final Map<String, String> fields; }
class UnexpectedException extends AppException { const UnexpectedException([super.message = 'unexpected error']); }
```
Mapeo desde transporte (factory en `AppException`):
```dart
factory AppException.fromResponse(int? status, dynamic body) {
  return switch (status) {
    401 => const UnauthorizedException(),
    404 => const NotFoundException(),
    400 => ValidationException(_msg(body), _fields(body)),
    null => const NetworkException(),   // sin red / timeout
    _   => const UnexpectedException(),
  };
}
```
Reglas verbatim que aplican:
- Los errores de dominio son subtipos de `AppException`. Widgets y notifiers manejan `AppException`, nunca excepciones crudas de plataforma.
- Los errores de transporte se mapean a `AppException` en la capa de datos (networking), no en widgets.
- Los notifiers capturan y traducen a un estado `error`; `build` nunca lanza. Sin `try/catch` que silencie sin superficie.
- Toda acción async de UI tiene estado visible de loading, success y error. Sin fallos silenciosos.

**Estado (`state-management`, Riverpod) — patrón:**
```dart
// lib/shared/state/health_notifier.dart
class HealthNotifier extends StateNotifier<HealthState> {
  HealthNotifier(this._repo) : super(const HealthState());
  final HealthRepository _repo;

  Future<void> checkHealth() async {
    state = state.copyWith(status: HealthStatus.loading);
    try {
      await _repo.check();
      state = state.copyWith(status: HealthStatus.alive);
    } on AppException catch (e) {
      state = state.copyWith(status: HealthStatus.error, error: e.message);
    }
  }
}
```
Reglas verbatim que aplican: estado compartido en `StateNotifier` con estado inmutable + `copyWith`; un provider por pieza de estado/servicio; servicios/repos vía providers (DI, no `new`); widgets `ConsumerWidget`/`ConsumerStatefulWidget`; side-effects vía `ref.listen`, `build` puro; el notifier expone métodos de intención (`checkHealth`), no setters crudos.

**Config de entorno (`env-config`, flutter_dotenv) — patrón:**
```dart
// lib/shared/config/env.dart
class Env {
  Env._(this.apiBaseUrl);
  final String apiBaseUrl;
  factory Env.fromDotenv() => Env._(_required('API_BASE_URL'));
  static String _required(String k) {
    final v = dotenv.env[k];
    if (v == null || v.isEmpty) throw StateError('missing env $k');
    return v;
  }
}
final envProvider = Provider<Env>((_) => throw UnimplementedError()); // override en main
```
```dart
// main.dart
Future<void> main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await dotenv.load(fileName: '.env');
  final env = Env.fromDotenv(); // fail-fast si falta API_BASE_URL
  runApp(ProviderScope(overrides: [envProvider.overrideWithValue(env)], child: const App()));
}
```
Reglas verbatim que aplican: config cargada y validada **una vez** al arranque (fail-fast); la app consume el `Env` tipado, nunca `dotenv.env` directo. `.env` es git-ignored; un `.env.example` commiteado documenta cada clave con placeholders.

**Navegación (`go_router`) — patrón:** una tabla de rutas única en `shared/router/`; paths como constantes en `routes.dart`; `errorBuilder` para 404. En esta story basta una ruta raíz `'/'` que apunta a `HealthScreen` (sin guards de sesión, ya que no hay auth — ADR-003).

**Theming (`theming`) — patrón:** todos los colores, estilos de texto y spacing vienen de tokens en `shared/theme/` (sin `Color(0x...)` ni `EdgeInsets.all(13)` inline). Naming semántico: `AppColors.success`, `AppColors.error` (justo lo que la pantalla de health necesita para distinguir vivo/error). `AppTheme.dark()` (o light) se construye una vez y se pasa a `MaterialApp.router`.

**Testing (`testing`, mocktail) — reglas:** `test/` espeja `lib/`; dependencias fakeadas overrideando providers en `ProviderContainer`/`ProviderScope`; mocks con mocktail (`when`/`verify`), `registerFallbackValue` para tipos custom; helper `pump_app.dart` envuelve en `ProviderScope` + `MaterialApp`; tests deterministas (sin red real); `addTearDown(container.dispose)`. Aserciones sobre comportamiento visible al usuario en widget tests.

### Task 3: ApiClient (http) + modelo de errores de dominio (AppException)

**Status:** Completed

#### Description
Implementar el `ApiClient` sobre `http.Client` (base URL desde `Env`, timeout explícito, `getJson`) y el modelo de errores de dominio `AppException` (sealed + subtipos) con el factory `AppException.fromResponse`. El `ApiClient` mapea status ≥ 400 y errores de transporte (sin red / timeout) a `AppException`. Provisto vía `apiClientProvider`. Da soporte a CA-2 (request a `/health`) y CA-3 (mapeo de errores).

#### Acceptance Criteria
1. `AppException` sealed con subtipos `NetworkException`, `NotFoundException`, `UnauthorizedException`, `ValidationException`, `UnexpectedException`; factory `AppException.fromResponse(int? status, dynamic body)` con el switch de la convención (`null` → `NetworkException`).
2. `ApiClient.getJson(path)` hace `GET ${baseUrl}${path}` con timeout explícito; decodifica body vacío de forma segura; si `statusCode >= 400` lanza `AppException.fromResponse(status, body)`.
3. Errores de transporte (`SocketException`, `TimeoutException`) se capturan y se traducen a `NetworkException` (status `null`).
4. `apiClientProvider` (`Provider<ApiClient>`) construye el cliente con `baseUrl: ref.read(envProvider).apiBaseUrl` y `timeout` explícito (p. ej. 15s).
5. El `ApiClient` no loguea respuestas completas ni headers.

#### Technical Details

**Architectural Context References:**
- Seguir la convención `networking` (patrón `ApiClient` — adaptar `getJson` desde el `postJson` de referencia) y `error-handling` (modelo `AppException` + `fromResponse`).
- ADR-001: el shape de error `{ "error": { code, message } }` se mapea en `ValidationException` (400) para reutilizar en FG-2; para `/health` solo importa 200 vs ≥400 vs sin-red.

**Files to Create/Modify:**
- `lib/shared/services/api_client.dart`
- `lib/shared/errors/app_exception.dart`
- `lib/shared/providers/api_client_provider.dart` (o `apiClientProvider` dentro de `api_client.dart`)

**Implementation Notes:**
- `http` no tiene interceptores: el timeout (`.timeout(...)`), el armado de URL y el mapeo de errores viven en el `ApiClient`.
- Envolver la llamada en `try/catch` para `SocketException`/`TimeoutException` → `NetworkException`.
- `AppException` es `sealed` para permitir `switch` exhaustivo en consumidores.

**Dependencies:**
- Externas: `http`, `flutter_riverpod`.
- Internas: `envProvider` (T2).
