**Estrategia de errores** (de `error-handling`, auto-incluida por `http-server`):

### ADRs Relevantes

#### ADR-001: Stack y arquitectura cliente-servidor

- **DecisiÃ³n:** atmosphere-api en Go 1.26.4 con router stdlib `net/http`, `encoding/json`; servicio HTTP stateless. Sin autenticaciÃ³n en v1; CORS habilitado.
- **Implementation Rules (verbatim):**
  - La API MUST usar **Go 1.26.4** y router **stdlib `net/http`** (ServeMux con patrones mÃ©todo+path). NO usar Gin, Echo, Fiber ni chi.
  - SerializaciÃ³n MUST ser `encoding/json` (stdlib).

  - El frontend MUST ser Flutter (Dart) con **Riverpod** (estado), **`http`** (cliente) e **`intl`** (formato). El frontend NO calcula: solo presenta lo que devuelve la API.
  - La comunicaciÃ³n MUST ser HTTP/REST JSON sÃ­ncrona; no hay procesos async ni mensajerÃ­a.

#### ADR-003: Servicio stateless (sin persistencia)

- **DecisiÃ³n:** atmosphere-api es stateless: sin base de datos, sin disco, sin sesiÃ³n.
- **Implementation Rules (verbatim):**
  - `atmosphere-api` MUST ser stateless: NO base de datos, NO persistencia en disco, NO estado de sesiÃ³n entre requests.
