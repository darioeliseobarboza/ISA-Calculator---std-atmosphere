# Story Plan: S-005 (atmosphere-app) - Pantalla de cálculo + presentación doble unidad

## Status

**Status:** Completed

## Referencia de la Story

- **Story:** [S-005](../stories/S-005.pantalla-calculo-presentacion.md)
- **Servicio:** atmosphere-app
- **Resumen:** Implementar el módulo `calculator` de atmosphere-app: una pantalla donde el usuario ingresa una altitud y su unidad (m/ft), se valida el formato, se llama a `POST /v1/calculate` y se presentan las 6 magnitudes ISA en SI e imperial (5 cifras significativas, notación científica para μ, ν y P/ρ) más los 5 relativos adimensionales y la altitud eco en `{m, ft}`, con estados de carga, error de validación y error de conexión, todo en español. La pantalla `calculator` pasa a ser la vista raíz, reemplazando la pantalla provisional de health (FG-1).

## Contexto Arquitectónico

> Esta sección es autocontenida: copia el contenido relevante de la documentación técnica. El implementador no debería necesitar abrir otros archivos del repo de producto.

### Tech Stack

- **Lenguaje:** Dart (Flutter, última estable; pinneado en `pubspec.yaml` `environment: sdk` y CI). Sound null safety en todo el código.
- **Estado / DI:** Riverpod (`flutter_riverpod`) — `StateNotifier` + providers; Riverpod también es el mecanismo de DI (no `get_it`).
- **Networking:** `http` (cliente oficial Dart) envuelto en un `ApiClient` (ADR-001 reemplaza el default de catálogo `dio`).
- **i18n / formato:** `intl` (`NumberFormat`) + ARB (`flutter gen-l10n`). App en español (`es`) en v1.
- **Navegación:** `go_router`.
- **Config:** `flutter_dotenv` (`Env` tipado, fail-fast; `API_BASE_URL`).
- **Modelos:** convención `models-serialization` (freezed + json_serializable para modelos nuevos; el app existente define modelos a mano — variante aceptable para modelos chicos).
- **Testing:** `flutter_test` + `mocktail` (mocks sin codegen); overrides de providers en `ProviderScope`/`ProviderContainer`.

### Arquitectura del Servicio

**Estructura feature-first + shared (convención `_base`):**

```
lib/
├── main.dart                  # bootstrap: dotenv, Env, ProviderScope, MaterialApp.router
├── screens/                   # una carpeta por feature
│   └── {feature}/
│       ├── {feature}_screen.dart
│       └── widgets/           # widgets usados solo por este feature
└── shared/                    # cross-cutting, reusado entre features
    ├── models/                # modelos de dominio / DTOs
    ├── providers/             # providers Riverpod (wiring)
    ├── state/                 # StateNotifiers + clases de estado
    ├── services/              # api_client.dart (networking)
    ├── theme/                 # design system (tokens)
    ├── i18n/ (o l10n/)        # localización (ARB)
    ├── format/                # number_format.dart (helpers intl para resultados ISA)
    ├── errors/                # app_exception.dart
    ├── config/                # env.dart
    ├── router/                # app_router.dart + routes.dart
    ├── utils/                 # helpers puros
    └── widgets/               # componentes UI reusables
```

- **Networking (repository + ApiClient):** widgets/notifiers dependen de una **interfaz de repositorio**, nunca de `http.Client`/`ApiClient` directamente. Un repositorio por dominio, provisto vía Riverpod. El `ApiClient` centraliza base URL, timeout explícito, headers y mapeo de errores (el paquete `http` no tiene interceptors).
- **Estado:** estado compartido en `StateNotifier`; clase de estado inmutable con `copyWith`. El notifier expone métodos de intención (no setters). Side-effects (navegación/snackbars) vía `ref.listen`, nunca en `build`.
- **Manejo de errores:** errores de dominio son subtipos de `AppException` (sealed). El mapeo transporte→dominio ocurre en la capa de datos (`ApiClient`), no en widgets. Los notifiers atrapan `AppException` y lo ponen en un campo `error` del estado; `build` nunca lanza. Toda acción async tiene estados visibles de loading, success y error.
- **Logging:** nunca loguear respuestas completas ni headers de auth.
- **Testing:** `test/` espeja `lib/`. Dependencias falseadas por override de providers. Mocks con `mocktail` (`when`/`verify`); `registerFallbackValue` para tipos custom. Tests deterministas (sin red real). Aserciones sobre comportamiento visible (texto, presencia de widgets) en widget tests. CI corre `flutter analyze` + `flutter test --coverage` y el analyzer debe quedar limpio.

### Relevant ADRs

#### ADR-001: Stack y arquitectura cliente-servidor

- **Decisión:** atmosphere-app es Flutter (Dart) con Riverpod (estado), `http` (cliente) e `intl` (formato). Comunicación HTTP/REST JSON síncrona request/response. Sin autenticación en v1.
- **Implementation Rules (verbatim, las que aplican al frontend):**
  - El frontend MUST ser Flutter (Dart) con **Riverpod** (estado), **`http`** (cliente) e **`intl`** (formato). El frontend NO calcula: solo presenta lo que devuelve la API.
  - La comunicación MUST ser HTTP/REST JSON síncrona; no hay procesos async ni mensajería.
  - Las rutas MUST llevar prefijo de versión **`/v1/`** (el endpoint es `POST /v1/calculate`).
  - Respuestas HTTP: `200` éxito; `400` validación con cuerpo `{ "error": { "code": string, "message": string } }`, con `code` ∈ `outOfRange` | `invalidInput` | `invalidStep`.
  - Los identificadores JSON MUST ir en **inglés** (campos/enums `camelCase`), idénticos a las entidades del dominio (`geopotentialAltitude`, `dynamicViscosity`, `relativeErrorPct`, …).

#### ADR-002: Unidad canónica ft + salida SI + imperial simultánea

- **Decisión:** la entrada lleva `altitudeUnit` (`m`/`ft`, default `ft`); cada magnitud absoluta se devuelve como `{si, imperial}`; los relativos son adimensionales (número único); la altitud se eco-devuelve en `{m, ft}`. El request NO incluye campo de sistema de unidades.
- **Implementation Rules (verbatim, las que aplican al frontend):**
  - `altitudeUnit` MUST ser enum `m` | `ft`, default `ft`.
  - Cada magnitud **absoluta** (`temperature`, `pressure`, `density`, `dynamicViscosity`, `kinematicViscosity`, `speedOfSound`) MUST devolverse como objeto `{ "si": number, "imperial": number }`.
  - Los **relativos** (`theta`, `delta`, `sigma`, `speedOfSoundRatio`, `viscosityRatio`) MUST ser un número único (adimensional).
  - Unidades: **SI** = K, Pa, kg/m³, Pa·s, m²/s, m/s; **imperial** = °R, lbf/ft² (psf), slug/ft³, slug/(ft·s), ft²/s, ft/s.
  - La respuesta MUST incluir `input.geopotentialAltitude` como `{ "m": number, "ft": number }`.
  - El request MUST NOT incluir un campo de sistema de unidades (la respuesta trae ambos).

#### ADR-005: Precisión numérica (presentación)

- **Decisión:** sin redondeo interno; el redondeo (5 cifras significativas) es **solo de presentación**.
- **Implementation Rules (verbatim, las que aplican al frontend):**
  - El redondeo es SOLO de presentación: **5 cifras significativas**, con notación científica para μ, ν y P/ρ cuando corresponda.
  - (El cómputo en `float64` y las constantes ocurren en la API; el frontend NO calcula — ADR-001.)

> **Nota:** ADR-003 (servicio stateless) y ADR-004 (deployment Docker) no imponen reglas de implementación sobre el código de este módulo del frontend; ADR-003 confirma que no hay persistencia (sin historial en v1).

### API Context

Contrato `atmosphere-api.yaml` — endpoint consumido por esta story: **`POST /v1/calculate`**.

**Request (`CalculationRequest`):**

```json
{
  "geopotentialAltitude": 16404,
  "altitudeUnit": "ft"
}
```

- `geopotentialAltitude` — `number`, **required**. Altitud geopotencial en la unidad de `altitudeUnit`.
- `altitudeUnit` — `string` enum `[m, ft]`, **default `ft`**.
- `tableStep` — `number`, default `1000`. **NO se envía en FG-2** (solo aplica a la interpolación, FG-3). Esta story NO incluye `tableStep` en el request.

**Response `200` (`CalculationResponse` — recorte FG-2: solo `input` + `results.analytical`):**

```json
{
  "input": {
    "geopotentialAltitude": { "m": 5000.0, "ft": 16404 },
    "altitudeUnit": "ft"
  },
  "results": {
    "analytical": {
      "method": "analytical",
      "temperature":       { "si": 255.65, "imperial": 460.17 },
      "pressure":          { "si": 54020,  "imperial": 1128.1 },
      "density":           { "si": 0.73643, "imperial": 0.0014290 },
      "dynamicViscosity":  { "si": 1.6286e-5, "imperial": 3.401e-7 },
      "kinematicViscosity":{ "si": 2.2117e-5, "imperial": 2.381e-4 },
      "speedOfSound":      { "si": 320.55, "imperial": 1051.7 },
      "theta": 0.8872,
      "delta": 0.5331,
      "sigma": 0.6009,
      "speedOfSoundRatio": 0.9420,
      "viscosityRatio": 0.9098
    }
  }
}
```

- `input.geopotentialAltitude` — objeto `{ m: number, ft: number }` (`AltitudeValue`).
- `input.altitudeUnit` — `string` enum `[m, ft]`.
- `input.tableStep` — **omitido en FG-2** (presente recién en FG-3).
- `results.analytical` — `AtmosphericResult`, **required**.
- `results.interpolation` — **omitido en FG-2** (FG-3).
- `comparison`, `table` — **omitidos en FG-2** (FG-3). El contrato es escalonado y aditivo: el mismo endpoint se completa en FG-3 sin breaking change, por eso son opcionales.

**`AtmosphericResult` (campos exactos):**
- `method` — `string` enum `[analytical, interpolation]`.
- Absolutas (`MagnitudeValue` = `{ si: number, imperial: number }`): `temperature`, `pressure`, `density`, `dynamicViscosity`, `kinematicViscosity`, `speedOfSound`.
- Relativos (number adimensional): `theta` (T/T₀), `delta` (P/P₀), `sigma` (ρ/ρ₀), `speedOfSoundRatio` (a/a₀=√θ), `viscosityRatio` (μ/μ₀).

**Response `400` (`ErrorResponse`):**

```json
{ "error": { "code": "outOfRange", "message": "geopotentialAltitude out of range (0–36089 ft ≈ 0–11000 m)" } }
```

- `error.code` — `string` enum `[outOfRange, invalidInput, invalidStep]`.
- `error.message` — `string`.

**Autenticación:** ninguna (ADR-001). Headers: `Content-Type: application/json`. Base URL desde `Env.apiBaseUrl`.

### Flow Context

Flujo `calculate` (`docs/flows/calculate.md`). Esta story implementa el lado app de la **rama analítica** (Pasos 1 y 3 del flujo). No reescribe el documento de flujo.

**Paso 1 — Usuario envía la solicitud de cálculo** (`atmosphere-app` → `atmosphere-api`, REST):
- Método: `POST`, Endpoint: `/v1/calculate`.
- Body: `{ geopotentialAltitude, altitudeUnit }` (sin `tableStep` en FG-2).
- `atmosphere-app` valida solo el **formato (numérico)** antes de enviar; la validación de **rango es autoritativa en la API** (Paso 2). Esto significa: el campo no numérico se bloquea client-side; el fuera-de-rango lo decide la API (400 `outOfRange`).

**Paso 3 — Respuesta al cliente** (`atmosphere-api` → `atmosphere-app`, REST):
- `200` con `{ input, results }` (en FG-2 sin `comparison`/`table`/`interpolation`).
- Acción del frontend: renderiza cada magnitud en SI e imperial con formato/precisión vía `intl` (5 cifras significativas).

**Manejo de Errores (tabla del flujo, lado app):**

| Error | Código | Comportamiento de la app |
|-------|--------|--------------------------|
| Entrada no numérica / vacía (client-side) | — | Bloquea el envío y marca el campo |
| Altitud fuera de `0–36089 ft` | `400 outOfRange` | Muestra el mensaje; sin resultados |
| Entrada no parseable en la API | `400 invalidInput` | Muestra el mensaje |
| Sin conectividad / API no responde | — (red) | Muestra error de conexión (no calcula offline); conserva la entrada |
