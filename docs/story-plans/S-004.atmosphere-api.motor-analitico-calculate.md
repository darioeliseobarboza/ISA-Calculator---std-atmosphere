- **`cmd/atmosphere-api/`**: solo entry point y wiring. La lógica del motor vive en `internal/calculation` y `internal/units`.
- **`internal/{module}/`**: dominio organizado por capacidad (un paquete por módulo). Esta story crea `calculation` y `units` (declarados en `manifest.yaml`).
- Wiring **manual y explícito** en `cmd/`. Sin framework DI. Constructores devuelven valor construido o `error` (nunca `panic` por mala config).
- **Accept interfaces, return structs.** El handler depende de un `Service` (el motor); para tests, el handler puede ejercerse con el `Service` real (cómputo puro, sin I/O) — no hace falta mock.

**Patrón handler (de `http-server`)** — el handler solo decodifica, valida, llama al service y escribe:

```go
// internal/calculation/handler.go
func (h *Handler) calculate(w http.ResponseWriter, r *http.Request) {
	var in CalculationRequest
	if err := json.NewDecoder(r.Body).Decode(&in); err != nil {
		writeError(w, r, errs.NewValidation("invalid body", nil)) // -> invalidInput / 400
		return
	}
	res, err := h.svc.Calculate(r.Context(), in)
	if err != nil {
		writeError(w, r, err) // mapeado por la convención error-handling
		return
	}
	WriteJSON(w, http.StatusOK, res)
}
```

**Patrón de rutas (de `http-server`)** — un `Routes`/`Mount` por módulo, montado central:

```go
// internal/calculation/handler.go
func (h *Handler) Routes(mux *http.ServeMux) {
	mux.HandleFunc("POST /v1/calculate", h.calculate)
}
// internal/http/routes.go (MODIFICAR para recibir el handler)
func Mount(calc *calculation.Handler) func(*http.ServeMux) {
	return func(mux *http.ServeMux) { calc.Routes(mux) }
}
```

> NOTA de integración: `Mount` hoy tiene firma `func Mount(_ *http.ServeMux)` y se pasa a `NewServer` como `apihttp.Mount`. Para FG-2 se cambia a `Mount(calc *calculation.Handler) func(*http.ServeMux)` y `main.go` pasa `apihttp.Mount(calcHandler)`. `NewServer` NO cambia (recibe el `mount` ya construido). El paquete `internal/http` deberá importar `internal/calculation` (sin ciclo: calculation no importa http).

**Estrategia de errores (de `error-handling`, auto-incluida por `http-server`):**

- Errores de dominio **tipados** (`*errs.Error` con `Kind`) o sentinela. Nunca `errors.New` "pelado" que el caller deba ramificar.
- Wrap con `%w` al cruzar capas; inspección con `errors.Is`/`errors.As`, nunca string matching.
- Forma canónica del error en el borde: `{ "error": { "code": string, "message": string } }`.
- **DESVIACIÓN del servicio (NORMATIVA — fijada por ADR-001, no es un error):** la convención genérica `error-handling` usa códigos `SCREAMING_SNAKE` (`VALIDATION_ERROR`, etc.). En **atmosphere-api** los códigos públicos del endpoint `POST /v1/calculate` son **`outOfRange` | `invalidInput` | `invalidStep`** (camelCase), definidos en ADR-001 y en el enum `ErrorResponse.error.code` del OpenAPI. En FG-2 solo se emiten **`outOfRange`** e **`invalidInput`**; `invalidStep` queda en el enum pero **inactivo** (llega en FG-3 con la tabla). El `code` interno `INTERNAL_ERROR` (500) del `recoverer` se mantiene (es el fallo no esperado, no un error de dominio del contrato `calculate`).
- En consecuencia, la `Kind`/constructor del paquete `errs` debe mapear a estos códigos camelCase en el borde. Recomendación de modelado (ver Task 1): un `errs.Error{ Code string; Message string; ... }` con constructores `errs.NewOutOfRange(msg)` y `errs.NewInvalidInput(msg)` que ya fijan el `code` público y el status `400`. El `writeError` (transporte) lee `Code` y status del error tipado; para errores no tipados responde `500 INTERNAL_ERROR`.
- Loguear un error **una sola vez**, en el borde (no log-and-return por capa). Los errores de dominio esperados (`outOfRange`, `invalidInput`) se loguean a `warn` sin stack; lo inesperado a `error`.

**Logging (de `logging` — slogx):** un logger por servicio, construido en `main` (S-001), inyectado. Correlación por request con `RequestArgs(ctx)`. Mensajes cortos/estáticos; datos en args key-value. El motor de cálculo no necesita loguear en el happy path; el borde HTTP loguea el resultado/erro una vez.

**Convención de testing (de `testing`):**

- Tests table-driven (varios casos del mismo comportamiento). Paquete black-box `package {pkg}_test` por defecto.
- `require` para precondiciones (corta el test), `assert` para checks independientes.
- Lógica de dominio (fórmulas, conversiones) se testea **en aislamiento** (cómputo puro, sin red/DB).
- HTTP se cubre con test de integración in-process (`httptest`); gatear integración con `testing.Short()`.
- `require.ErrorIs`/`ErrorAs` para identidad de errores, nunca comparar strings. `-race` en CI.
- Para floats: comparar con tolerancia (`assert.InEpsilon`/`InDelta`), nunca igualdad exacta.

### ADRs Relevantes

#### ADR-001: Stack y arquitectura cliente-servidor

- **Decisión:** atmosphere-api en Go 1.26.4 con router stdlib `net/http`, `encoding/json`; servicio HTTP stateless. Sin autenticación en v1; CORS habilitado.
- **Implementation Rules (verbatim, las relevantes a esta story):**
  - La API MUST usar **Go 1.26.4** y router **stdlib `net/http`** (ServeMux con patrones método+path). NO usar Gin, Echo, Fiber ni chi.
  - Serialización MUST ser `encoding/json` (stdlib).
  - Las rutas MUST llevar prefijo de versión **`/v1/`** (aplica a `POST /v1/calculate`).
  - La comunicación MUST ser HTTP/REST JSON síncrona; no hay procesos async ni mensajería.
  - Respuestas HTTP: `200` éxito; `400` validación con cuerpo `{ "error": { "code": string, "message": string } }`, con `code` ∈ `outOfRange` | `invalidInput` | `invalidStep`.
  - Los identificadores JSON MUST ir en **inglés** (tipos `PascalCase`, campos/enums `camelCase`), idénticos a las entidades del dominio.

#### ADR-002: Unidad canónica de cálculo en pies (ft) y salida SI + imperial simultánea

- **Decisión:** unidad canónica interna = ft; entrada en m|ft (default ft); validación `0–36089 ft`; salida de cada absoluta en `{si, imperial}`; relativos adimensionales; eco de altitud en `{m, ft}`.
- **Implementation Rules (verbatim):**
  - La unidad canónica interna de altitud MUST ser **ft**; la entrada se normaliza a ft con `1 ft = 0.3048 m` (exacto) antes de validar/calcular.
  - `altitudeUnit` MUST ser enum `m` | `ft`, default `ft`.
  - Validación MUST ser `0 ≤ h ≤ 36089 ft`; fuera de rango → `400` `outOfRange`.
  - Cada magnitud **absoluta** (`temperature`, `pressure`, `density`, `dynamicViscosity`, `kinematicViscosity`, `speedOfSound`) MUST devolverse como objeto `{ "si": number, "imperial": number }`.
  - Los **relativos** (`theta`, `delta`, `sigma`, `speedOfSoundRatio`, `viscosityRatio`) MUST ser un número único (adimensional).
  - Unidades: **SI** = K, Pa, kg/m³, Pa·s, m²/s, m/s; **imperial** = °R, lbf/ft² (psf), slug/ft³, slug/(ft·s), ft²/s, ft/s.
  - La respuesta MUST incluir `input.geopotentialAltitude` como `{ "m": number, "ft": number }`.
  - El request MUST NOT incluir un campo de sistema de unidades (la respuesta trae ambos).

#### ADR-003: Servicio stateless (sin persistencia)

- **Decisión:** atmosphere-api es stateless: sin base de datos, sin disco, sin sesión.
- **Implementation Rules (verbatim, relevantes):**
  - `atmosphere-api` MUST ser stateless: NO base de datos, NO persistencia en disco, NO estado de sesión entre requests.
  - El servicio MUST poder escalar horizontalmente sin estado compartido (cualquier instancia responde cualquier request).
  - (La `ISATable` en memoria y el historial NO aplican a FG-2: no hay tabla ni interpolación en esta story.)

#### ADR-005: Precisión numérica (float64 + constantes exactas ISA 1976 derivadas) — **núcleo de esta story**

- **Decisión:** cómputo en `float64`; constantes base exactas ISA 1976/ICAO; valores derivados (R, ρ₀, a₀, μ₀, exponente n) calculados **en runtime**; sin redondeo interno (el redondeo es solo de presentación, 5 cifras significativas); la tabla UTN es cross-check, no el techo de precisión.
- **Implementation Rules (verbatim):**
  - Todo el cómputo MUST hacerse en `float64`.
  - Constantes base (exactas) MUST ser: `R* = 8.31432 J/(mol·K)`, `M₀ = 0.0289644 kg/mol`, `T₀ = 288.15 K`, `P₀ = 101325 Pa`, `L = −0.0065 K/m` (= `−0.0019812 K/ft`), `g₀ = 9.80665 m/s²`, `γ = 1.4`, `β = 1.458e-6 kg/(m·s·√K)`, `S = 110.4 K`.
  - Valores derivados MUST calcularse en runtime (NO hardcodear redondeados): `R = R*/M₀`, `ρ₀ = P₀/(R·T₀)`, `a₀ = √(γ·R·T₀)`, `μ₀ = β·T₀^1.5/(T₀+S)`, `n = g₀/(R·|L|)`.
  - MUST NOT sustituir las constantes ISA 1976 por valores físicos más nuevos (mantener coherencia con la ISA y la tabla de referencia).
  - MUST NOT redondear en el cálculo; el redondeo es SOLO de presentación: **5 cifras significativas**, con notación científica para μ, ν y P/ρ cuando corresponda.
  - Validación: el método analítico MUST coincidir con la tabla UTN (en ft) dentro de su tolerancia de redondeo (3 decimales) en `0–36.089 ft`.

> **Atención a las unidades de `L` y `n`:** la unidad canónica de altitud es **ft** (ADR-002), pero las constantes base son métricas. Hay dos caminos equivalentes y correctos: (a) trabajar el modelo de temperatura en SI (h en m, `L = −0.0065 K/m`) convirtiendo ft→m sólo para la fórmula de T y P; o (b) usar el gradiente en ft (`L = −0.0019812 K/ft`) y el exponente `n = g₀/(R·|L_m|)` (adimensional, NO cambia con la unidad de altura porque es un exponente sobre el ratio T/T₀). **El exponente `n` es adimensional y se calcula con `L` y `R` en SI.** La fórmula de presión usa el ratio de temperaturas, no la altura directa, así que es indistinto. El implementador DEBE elegir un camino y validar con la tabla UTN (CA-6). Recomendación: computar T con `L` en K/ft sobre la altura en ft (alineado a ADR-002), y P/ρ con el ratio T/T₀ y el exponente `n` adimensional.

**Fórmulas analíticas de la tropósfera (capa de gradiente, 0–36.089 ft / 0–11.000 m), `h` = altitud geopotencial:**

- Temperatura: `T = T₀ + L·h` (con `L` y `h` en unidades coherentes; `T` en K).
- Relativa de temperatura: `θ = T/T₀`.
- Presión relativa: `δ = (T/T₀)^n = θ^n`, con `n = g₀/(R·|L|)` (≈ 5.2559). `P = P₀·δ`.
- Densidad relativa: `σ = δ/θ = θ^(n−1)`. `ρ = ρ₀·σ` (o `ρ = P/(R·T)`).
- Velocidad del sonido: `a = √(γ·R·T)`; relativa `a/a₀ = √θ`.
- Viscosidad dinámica (Sutherland): `μ = β·T^1.5/(T+S)`; relativa `μ/μ₀`.
- Viscosidad cinemática: `ν = μ/ρ`.

### Contexto de API

#### `POST /v1/calculate` — etapa analítica (FG-2)

(de `docs/apis/atmosphere-api.yaml`, `operationId: calculate`)

- **Método + path:** `POST /v1/calculate` (con prefijo `/v1/` — ADR-001).
- **Auth:** ninguna (ADR-001). **Content-Type:** `application/json`.
- **Request body — `CalculationRequest`:**
  ```json
  {
    "geopotentialAltitude": 16404,   // number, REQUERIDO. En la unidad de altitudeUnit.

    "input": {
      "geopotentialAltitude": { "m": 5000.0, "ft": 16404 },   // AltitudeValue {m, ft}
      "altitudeUnit": "ft"

      "analytical": {                                          // AtmosphericResult
        "method": "analytical",

  ```

- **Archivos:** minúscula, una responsabilidad por archivo (`constants.go`, `troposphere.go`, `service.go`, `handler.go`, `dto.go`).
- **Test files:** `*_test.go`, paquete `{pkg}_test` para black-box.
- **Identificadores exportados:** `MixedCaps` (`Service`, `Calculate`, `NewService`). No exportados: `mixedCaps`. Acrónimos conservan caja (`ISA`, `URL`, `requestID`). El módulo `units`: p. ej. `FeetFromMeters`, `PascalToPSF`.

- **Identificadores JSON:** en **inglés**, `camelCase` en campos/enums; tipos `PascalCase` (ADR-001). Tags `json:"geopotentialAltitude"`, etc.
- **Errores:** wrap con `%w`; `context.Context` primer parámetro en funciones que cruzan capas (el `Service.Calculate(ctx, in)` lleva `ctx` por convención aunque el cómputo sea puro). `context.Background()` solo en `main`/tests.

- **Floats:** comparar en tests con tolerancia (`InEpsilon`/`InDelta`), nunca igualdad exacta.

DTOs en `dto.go`: `CalculationRequest { GeopotentialAltitude float64 \`json:"geopotentialAltitude"\`; AltitudeUnit string \`json:"altitudeUnit"\`; TableStep float64 \`json:"tableStep"\` }` (con default `altitudeUnit="ft"` aplicado al decodificar/normalizar; `tableStep` aceptado pero IGNORADO). Response DTOs: `MagnitudeValue {si, imperial}`, `AltitudeValue {m, ft}`, `AtmosphericResult {method, temperature, pressure, density, dynamicViscosity, kinematicViscosity, speedOfSound, theta, delta, sigma, speedOfSoundRatio, viscosityRatio}`, `CalculationResponse {input, results}` donde `input` lleva `geopotentialAltitude (AltitudeValue)` y `altitudeUnit`, y `results` lleva solo `analytical (AtmosphericResult)`. **Omitir** en el JSON: `input.tableStep`, `results.interpolation`, `comparison`, `table` (usar `omitempty`/no incluir los campos en FG-2).
2. `Service.Calculate(ctx context.Context, in CalculationRequest) (CalculationResponse, error)`: aplica default `altitudeUnit="ft"`; normaliza a ft; valida `0 ≤ h ≤ 36089 ft` → si falla, `errs.NewOutOfRange(...)`; si la entrada es no numérica/no parseable (NaN/Inf) → `errs.NewInvalidInput(...)`; calcula analítico; arma `{si, imperial}` y eco `{m, ft}`; `results.analytical.method == "analytical"`.
3. `Handler.calculate(w, r)`: decodifica el body; si el JSON es inválido o el campo requerido falta/no es número → `writeError(w, r, errs.NewInvalidInput(...))`; si no, llama al service y, ante error, `writeError(w, r, err)`; en éxito `WriteJSON(w, 200, res)`.
4. `Handler.Routes(mux)` registra `mux.HandleFunc("POST /v1/calculate", h.calculate)`. `internal/http/routes.go` `Mount` se modifica a `func Mount(calc *calculation.Handler) func(*http.ServeMux)` y monta `calc.Routes`. `main.go` construye `calculation.NewService()` + `calculation.NewHandler(svc, ...)` y pasa `apihttp.Mount(handler)` a `NewServer`.
5. `NewServer` NO se modifica. El `POST` ya está advertido en CORS (S-001).
