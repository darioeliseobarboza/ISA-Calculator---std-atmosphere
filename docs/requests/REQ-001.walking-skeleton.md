# REQ-001: Esqueleto / Walking Skeleton (FG-1)

## Requerimiento Original

> **Feature Group 1: Esqueleto / Walking Skeleton**
>
> Infraestructura mínima y "sistema vivo" end-to-end, sin features de negocio. Se crean
> las dos bases de código —`atmosphere-api` (Go + `net/http`, stateless) y `atmosphere-app`
> (Flutter + Riverpod + `http`)— con un endpoint trivial `GET /health` en la API y una
> pantalla en el frontend que lo consume y muestra el estado. Se configura **CORS** (para el
> target Web) y el **empaquetado** de los tres targets: Dockerfile para Web y documentación
> de build de escritorio (Windows/Linux).
>
> Valor entregado: el sistema está "vivo" end-to-end: el frontend, en cualquiera de los tres
> targets, llega a la API y muestra que responde.

**Fuente:** PRD — `docs/prd/feature-groups.md` (Feature Group 1)
**Fecha de recepción:** 2026-06-15

## Clarificaciones

### Funcionales
- **P:** ¿La pantalla de health es definitiva o provisional?
- **R:** Provisional — es la prueba de vida del esqueleto; FG-2 la reemplaza por la UI real de la calculadora.

### Alcance
- **P:** ¿FG-1 incluye algún cálculo ISA o feature de negocio?
- **R:** No. FG-1 es estrictamente infraestructura + `GET /health` + build/empaquetado. El cálculo llega en FG-2/FG-3.

### Negocio
- **P:** ¿Impacta entidades del dominio?
- **R:** No. `/health` no es una entidad de dominio; FG-1 no introduce ni modifica entidades.

## Requerimientos Funcionales

1. `atmosphere-api` expone `GET /health` que responde un estado saludable (`status: "ok"` + `timestamp`).
2. `atmosphere-app` consulta `/health` al abrir y muestra si la API está **disponible** ("sistema vivo").
3. El frontend funciona en los tres targets: Windows, Linux y Web.
4. La llamada del target **Web** a la API es cross-origin con **CORS** habilitado (sin error de CORS).
5. Si la API no responde / no hay conectividad, la app muestra un **estado de error de conexión** (no "vivo").
6. `atmosphere-app` se compila/empaqueta para los tres targets (Web vía contenedor con Dockerfile; escritorio documentado).

## Criterios de Aceptación

1. DADO `atmosphere-api` corriendo
   CUANDO consulto `GET /health`
   ENTONCES responde `200` con un estado saludable (`status: "ok"` + `timestamp`)

2. DADO `atmosphere-app` corriendo en cualquiera de los 3 targets (Windows, Linux, Web)
   CUANDO abro la app
   ENTONCES consulta `/health` y muestra que la API está disponible ("sistema vivo")

3. DADO el target Web servido desde su contenedor y la API en otro origen
   CUANDO la app web llama a `/health`
   ENTONCES la llamada cross-origin funciona (CORS habilitado), sin error de CORS

4. DADO `atmosphere-app` corriendo
   CUANDO la API no responde / no hay conectividad
   ENTONCES la app muestra un estado de error de conexión (no "vivo")

5. DADO el monorepo
   CUANDO se buildea/empaqueta `atmosphere-app` para Windows, Linux y Web
   ENTONCES los tres targets compilan y la app corre (Web vía contenedor con Dockerfile)

### Cambios de API

#### Endpoints Existentes a Modificar
Ninguno.

#### Nuevos Endpoints Necesarios
- **`GET /health`** — ya definido en `docs/apis/atmosphere-api.yaml` (`operationId: healthCheck`). FG-1 lo **implementa tal cual**: `200 { "status": "ok", "timestamp": <RFC3339> }`. No requiere cambios al spec. `POST /v1/calculate` **no** se implementa en FG-1.

### Cambios de Base de Datos
Ninguno (servicio stateless, ADR-003).

### Flujo de Interacción

#### Escenario: Sistema vivo (health check)

**Trigger:** el usuario abre `atmosphere-app` (cualquier target).

**Flujo:**
1. **[atmosphere-app]** `ApiClient` (http) → `GET /health` a `atmosphere-api`, base URL desde env (`API_BASE_URL`).
2. **[atmosphere-api]** responde `200 { "status": "ok", "timestamp": ... }` (sin auth; CORS habilitado para el origen web).
3. **[atmosphere-app]** muestra "API disponible" ("sistema vivo").

**Outcome:** el sistema está vivo end-to-end en los 3 targets.

**Manejo de errores:**
- Sin conectividad / timeout / status ≥ 400 → la app muestra **estado de error de conexión** (no "vivo").

### Consideraciones Técnicas

#### Riesgos
- **CORS mal configurado** para el target Web → la app web no alcanza la API. **Mitigación:** `CORS_ALLOWED_ORIGINS` por env, probado vía `docker-compose.dev.yml`.
- **Drift dev/prod** del compose. **Mitigación:** patrón base + overrides (ADR-004).

#### Dependencias
- `atmosphere-app` depende del `GET /health` de `atmosphere-api`.
- El compose prod depende de la red externa `ingress-network` y del proxy (nginx-proxy + acme-companion).

#### Performance
- Trivial; `/health` es barato (liveness).

#### Seguridad
- Sin autenticación (ADR-001); solo **CORS** configurable. TLS lo termina el proxy en prod.

### Flujos Afectados

#### Flujos Existentes Modificados
- Ninguno. `calculate.md` (FG-2/3) no se toca en FG-1.

#### Nuevos Flujos
- **health-check** (trivial): `atmosphere-app → GET /health → atmosphere-api`. Por ser un ping de liveness, se deja implícito; no se crea un doc de flujo nuevo en v1 (el flujo principal sigue siendo `calculate`).
