# S-003: Orquestación Docker (compose dev/prod)

**Status:** Ready
**Created:** 2026-06-15
**Source:** REQ-001

## Historia de Usuario

**Como** desarrollador / operador,
**quiero** levantar el sistema con `docker-compose` (dev) y tener el compose de producción,
**para** correr y desplegar el "sistema vivo" end-to-end con paridad entre ambientes.

## Criterios de Aceptación

### CA-1: Entorno local con docker-compose

**DADO** el monorepo
**CUANDO** ejecuto `docker compose -f docker/docker-compose.yml -f docker/docker-compose.dev.yml up`
**ENTONCES** levantan `atmosphere-api` y `atmosphere-web` (build desde el path) con puertos expuestos en localhost, y la web alcanza la API (`/health`)

### CA-2: CORS end-to-end en dev

**DADO** el entorno dev levantado
**CUANDO** la app web (origen localhost) llama a `/health` de la API
**ENTONCES** la llamada cross-origin funciona (CORS configurado con el origen local)

### CA-3: Compose de producción

**DADO** `docker/docker-compose.prod.yml`
**CUANDO** lo reviso/uso
**ENTONCES** usa **imágenes del registry oficial**, conecta los servicios a la red externa `ingress-network`, **no** expone puertos al host, y cada servicio expuesto define `VIRTUAL_HOST`, `VIRTUAL_PORT` y `LETSENCRYPT_HOST`

### CA-4: Paridad base + overrides

**DADO** los compose
**CUANDO** comparo dev y prod
**ENTONCES** lo común vive una sola vez en `docker-compose.yml` (base) y cada override solo difiere en `build` vs `image`, red y exposición de puertos

### Flujo de Interacción

#### Escenario: Levantar el sistema en dev

**Trigger:** `docker compose -f docker/docker-compose.yml -f docker/docker-compose.dev.yml up`.

**Flujo:**
1. **[compose dev]** build de `atmosphere-api` y `atmosphere-web` desde el path del proyecto; puertos en localhost (p. ej. api `8080`, web `8081`).
2. **[atmosphere-web]** sirve el build de Flutter web; llama a la API por `API_BASE_URL`.
3. **[atmosphere-api]** responde `/health` con CORS para el origen local.

**Outcome:** sistema vivo end-to-end en local; mismo armado que prod salvo build/red/puertos.

**Manejo de errores:**
- `ingress-network` inexistente (prod) → documentar prerequisito (red externa + proxy).
- Drift dev/prod → mitigado por base + overrides.
