# S-004: Motor analítico ISA + POST /v1/calculate (analítico)

**Status:** Ready
**Created:** 2026-06-21
**Source:** REQ-002

## Historia de Usuario

**Como** consumidor de la API (la app o un cliente HTTP),
**quiero** un endpoint `POST /v1/calculate` que, dada una altitud geopotencial, calcule por método analítico los parámetros ISA de la tropósfera,
**para** obtener el ground-truth atmosférico (T, P, ρ, μ, ν, a y los relativos) en SI e imperial con exactitud verificable.

## Criterios de Aceptación

### CA-1: Cálculo analítico a altitud intermedia (en ft)

**DADO** `atmosphere-api` corriendo
**CUANDO** envío `POST /v1/calculate` con `{ "geopotentialAltitude": 16404, "altitudeUnit": "ft" }` (≈ 5.000 m)
**ENTONCES** responde `200` con `results.analytical` donde T ≈ 255,65 K (≈ 460,2 °R), P ≈ 54.020 Pa (≈ 1.128 psf), y ρ, μ, ν, a en `{si, imperial}`, más los relativos θ, δ, σ, a/a₀, μ/μ₀ como número único
**Y** la respuesta incluye `input.geopotentialAltitude = { "m": 5000.0, "ft": 16404 }`

### CA-2: Nivel del mar ISA (entrada en m)

**DADO** `atmosphere-api` corriendo
**CUANDO** envío `{ "geopotentialAltitude": 0, "altitudeUnit": "m" }`
**ENTONCES** obtengo T = 288,15 K, P = 101.325 Pa, ρ ≈ 1,2250 kg/m³, a ≈ 340,29 m/s (con su par imperial) y los relativos θ = δ = σ = a/a₀ = μ/μ₀ = 1
**Y** la altitud eco es `input.geopotentialAltitude = { "m": 0, "ft": 0 }`

### CA-3: Altitud fuera de rango → 400 outOfRange

**DADO** `atmosphere-api` corriendo
**CUANDO** envío `{ "geopotentialAltitude": 40000, "altitudeUnit": "ft" }` (> 36.089 ft)
**ENTONCES** la API responde `400` con `error.code = "outOfRange"` y no calcula

### CA-4: Entrada no parseable → 400 invalidInput

**DADO** `atmosphere-api` corriendo
**CUANDO** llega una entrada no numérica / no parseable en `geopotentialAltitude`
**ENTONCES** la API responde `400` con `error.code = "invalidInput"` y no calcula

### CA-5: Cada absoluta en doble unidad, relativos adimensionales

**DADO** un cálculo analítico exitoso
**CUANDO** inspecciono `results.analytical`
**ENTONCES** cada magnitud absoluta (temperature, pressure, density, dynamicViscosity, kinematicViscosity, speedOfSound) aparece como objeto `{ si, imperial }`
**Y** los relativos (theta, delta, sigma, speedOfSoundRatio, viscosityRatio) aparecen como número único adimensional
**Y** `results.analytical.method = "analytical"`

### CA-6: Cross-check contra la tabla UTN

**DADO** un conjunto de altitudes presentes en la tabla ISA de referencia (UTN, en ft)
**CUANDO** calculo por el método analítico
**ENTONCES** los valores coinciden con la tabla dentro de su tolerancia de redondeo (3 decimales) en `0–36.089 ft` (el valor exacto de tolerancia se fija en los tests)

### Cambios de API

#### Endpoints Existentes a Modificar

**`POST /v1/calculate`** (ref: `docs/apis/atmosphere-api.yaml`:65–92; schema `CalculationResponse`:120–156)

Contrato escalonado y aditivo: FG-2 entrega solo el método analítico. El schema se relaja en esta story (ver Step 5 / `docs/apis/atmosphere-api.yaml`) para soportar la etapa sin romper FG-3.

- **Request** — `CalculationRequest` tal cual (líneas 97–118): `geopotentialAltitude` (number, requerido), `altitudeUnit` (enum `m|ft`, default `ft`).
  - `tableStep` → **aceptado pero IGNORADO** en FG-2 (sin tabla/interpolación); se activa en FG-3.
- **Response 200** — subconjunto de `CalculationResponse`:
  - `input`: `{ geopotentialAltitude: AltitudeValue {m, ft}, altitudeUnit }`. `input.tableStep` → **omitido** en FG-2.
  - `results.analytical`: `AtmosphericResult` completo con `method: "analytical"`; absolutas como `MagnitudeValue {si, imperial}`; relativos como number.
  - `results.interpolation`, `comparison`, `table` → **omitidos en FG-2** (FG-3).
- **Cambio de schema (aplicado en esta story):**
  - `CalculationResponse.required`: quitar `comparison` y `table` (→ opcionales).
  - `results.required`: quitar `interpolation` (→ opcional).
  - Evolución aditiva: el **mismo endpoint** se completa en FG-3 sin breaking change.
- **Errores (`ErrorResponse`):** en FG-2 `error.code` toma solo **`outOfRange`** (altitud fuera de `0–36089 ft`) e **`invalidInput`** (no numérica / no parseable). `invalidStep` permanece en el enum pero **inactivo** hasta FG-3.

#### Nuevos Endpoints

Ninguno. `GET /health` se reusa tal cual (S-001), sin cambios.

### Cambios de Base de Datos

Ninguno. Servicio **stateless** (ADR-003); no existe `docs/db-schemas/`. La capa `troposphere` y las `PhysicalConstants` son datos en memoria/configuración, no persistidos.

### Flujo de Interacción

#### Escenario: Cálculo analítico exitoso (CA-1) — lado API

**Trigger:** llega un `POST /v1/calculate` con altitud y unidad.

**Flujo:**

1. **[atmosphere-api — http/middleware]**
   - Acción: CORS evalúa el origen (S-001); rutea al handler `calculate`.

2. **[atmosphere-api — units + calculation]**
   - Acción: normaliza `geopotentialAltitude` a **ft** (`1 ft = 0.3048 m`, ADR-002); valida `0 ≤ h ≤ 36089 ft`.
   - Acción: calcula **analytical** en `float64` (constantes ISA 1976 + derivadas en runtime, ADR-005): T, P, ρ, μ (Sutherland), ν=μ/ρ, a=√(γRT) y relativos θ, δ, σ, a/a₀, μ/μ₀.
   - Acción: convierte cada absoluta a `{si, imperial}` (módulo `units`); arma `input.geopotentialAltitude = {m, ft}`.

3. **[atmosphere-api]**
   - Output: `200 OK` `{ "input": {...}, "results": { "analytical": {...} } }`.

**Outcome:** la API devuelve T, P, ρ, μ, ν, a en SI e imperial + los relativos + la altitud de entrada en `{m, ft}`.

**Manejo de errores:**
- Altitud fuera de rango → `400 { "error": { "code": "outOfRange", "message": "…" } }` (CA-3).
- Entrada no parseable → `400 invalidInput` (CA-4).

### Flujos Afectados

- **calculate** (ref: `docs/flows/calculate.md`) — esta story implementa su **rama analítica** (Pasos 1–3 sin generación de `ISATable`, sin `interpolation` ni `comparison`, sin metadata `table`). No se reescribe el documento de flujo (describe el end-state de FG-3); la cobertura parcial de FG-2 queda registrada en REQ-002.

### Consideraciones Técnicas

**Seguridad:**
- Sin autenticación (ADR-001). CORS configurable (S-001). El cálculo nunca ocurre en el cliente. Sin datos sensibles ni persistencia.

**Performance:**
- NFR-P01: cálculo server-side < 100 ms (P95) por punto. Cómputo en memoria, trivial (sin tabla en FG-2).

**Dependencias:**
- Reusa la base de FG-1 (S-001): plumbing HTTP, config (dotenv), logging (slogx), CORS.
- Habilita S-005 (la app consume este endpoint).

**Exactitud (riesgo principal):**
- Riesgo de divergencia vs. la tabla de referencia por constantes incorrectas. Mitigación: constantes fijadas en ADR-005 + tests de cross-check contra la tabla UTN (`docs/references/`) en `0–36089 ft` (NFR-Q01, CA-6).
