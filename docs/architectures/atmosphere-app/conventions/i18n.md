# Internationalization (Flutter, intl)

User-facing strings are externalized into ARB files and resolved through the generated
`AppLocalizations` (`flutter gen-l10n`), and locale-aware number/date formatting uses
[intl](https://pub.dev/packages/intl). No literal user-facing text in widgets. Service-local
replacement of the catalog default (JSON + translation service): atmosphere-app uses `intl`,
both for strings and for the precise formatting of ISA results (NFR-U02).

## When to use

- Any app shipping in one or more languages. atmosphere-app ships in **Spanish** in v1, but strings are externalized so adding a language later is cheap.
- Whenever numbers/dates must be formatted locale-aware — central to this app (result precision).

## Package

```
intl                          # formatting (NumberFormat, DateFormat) + Intl
flutter_localizations (SDK)   # localization delegates
# tooling: flutter gen-l10n (ARB -> AppLocalizations), configured via l10n.yaml
```

## Structure

```
lib/l10n/
├── app_es.arb               # Spanish strings (default)
└── (app_en.arb, ...)        # future languages
l10n.yaml                    # gen-l10n config (template-arb-file, output-class)
lib/shared/format/
└── number_format.dart       # NumberFormat helpers for ISA results (sig figs, scientific)
```

Declare `generate: true` in `pubspec.yaml`; `AppLocalizations` is generated from the ARB files.

## How to use

### Resource files (ARB)

```json
// lib/l10n/app_es.arb
{
  "calc.title": "Calculadora ISA",
  "calc.altitude": "Altitud geopotencial",
  "calc.outOfRange": "Altitud fuera de rango (0–36089 ft)"
}
```

### Resolving a key

```dart
final l10n = AppLocalizations.of(context);
Text(l10n.calcTitle);
```

### Runtime language switch

The active `Locale` is a provider; changing it rebuilds the tree (see `state-management`).
v1 fixes the locale to `es`, but the mechanism (detect platform locale, allow override,
persist) is in place so adding a language needs no widget changes.

### Number & locale formatting (intl)

```dart
// lib/shared/format/number_format.dart — 5 significant figures, scientific when extreme
String formatSigFigs(num value, {int sig = 5}) {
  // NumberFormat from intl, locale 'es'; switch to scientific for very small/large magnitudes
  if (value != 0 && (value.abs() < 1e-3 || value.abs() >= 1e6)) {
    return NumberFormat.scientificPattern('es').format(value);
  }
  return NumberFormat('#,##0.#####', 'es').format(value);
}
```

Used to render T, P, ρ, μ, ν, a with 5 significant figures and scientific notation where appropriate (NFR-U02).

## Rules

- No user-facing literal strings in widgets. Every visible string is an ARB key resolved via `AppLocalizations`.
- Keys are namespaced by feature (`calc.title`, `calc.altitude`). Flat, descriptive, stable.
- Every key exists in **all** ARB files; a missing key falls back to the default language (`es`), never shows the raw key in production. (v1 has only `es`.)
- Interpolation uses **named ARB placeholders**, not string concatenation.
- Locale-aware formatting (numbers, dates) uses **`intl`** (`NumberFormat`/`DateFormat`), never manual formatting. ISA results use the shared `formatSigFigs` helper: 5 significant figures, scientific notation for extreme magnitudes (NFR-U02).
- Proper names/symbols are not translated; UI labels and messages are.
- Adding a language = adding an ARB file + `flutter gen-l10n`. No code changes in widgets.

## Variant

`slang` (type-safe keys via codegen) or raw `Intl.message` are acceptable. This app uses the
**ARB + gen-l10n** approach for compile-time-checked accessors plus `intl` formatting. The
rules (externalized strings, namespaced keys, full coverage, `intl` formatting) apply
identically. Document any deviation in `overview.md`.

## Integration with other conventions

- **state-management**: the active `Locale` is a provider; widgets watch it to rebuild on switch.
- **theming**: text styles come from the design system; i18n supplies the content and the number formatting.
